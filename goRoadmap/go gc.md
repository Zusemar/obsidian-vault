MARK

Go запускает **фоновые горутины GC** (`gcBgMarkWorker`), которые работают параллельно с твоим кодом.
```
         ┌─────────────────────────────────────────────┐
         │           GC Mark Phase (трицветная)        │
         └─────────────────────────────────────────────┘
                       │
             [STW #1]  │  мир остановлен
                       ▼
        ┌───────────────┐
        │  Найти roots  │  ← стеки, глобалы, регистры
        └───────┬───────┘
                │
                ▼
        ┌──────────────────────────────────────────────┐
        │  Concurrent Marking (фоновая маркировка)     │
        │                                              │
        │  gcBgMarkWorker помечает серые → чёрные      │
        │  write barrier защищает новые ссылки          │
        │  mutator assist помогает при аллокациях       │
        └──────────────────────────────────────────────┘
                       │
                [STW #2] короткая пауза
                       ▼
        ┌───────────────┐
        │  Finish mark  │
        └───────────────┘
                       │
                       ▼
               → переход в Sweep phase

```

STW
 **STW #1 — начало маркировки (Start of Mark phase)**

**Момент:** когда Go решает, что пора запустить новый GC (heap вырос на `GOGC%`).
Останавливает все горутины, чтобы получить _чистое_ состояние стека.
- Помечает все корневые объекты (`roots`):
    - стеки всех горутин;
    - глобальные переменные;
    - регистры;
    - системные структуры рантайма.
- Запускает фоновую фазу **mark** (concurrent marking).
- Возобновляет выполнение всех горутин.

 **STW #2 — конец маркировки (Mark Termination)**

**Момент:** когда фоновая маркировка почти закончена,  
но остаётся немного "висячих" объектов (например, ссылки, изменённые в процессе).

**Что делает:**

1. Снова останавливает все горутины.
2. Завершает маркировку — промечает остатки живых объектов.
3. Переключает GC в фазу **sweep**.
4. Возобновляет мир.

**Зачем:**  
Чтобы гарантировать, что **к моменту начала очистки все живые объекты действительно промечены**.  
Без этой паузы GC мог бы случайно освободить ещё используемый объект.

SWEEP 
- Проходит по всем span'ам (небольшим блокам памяти в heap).
    
- Проверяет, какие объекты в них были **помечены как живые**.
    
- Те, что не были помечены (белые), **освобождаются**.
    
- Освобождённые span’ы возвращаются в пул свободных.

момент sweep определяется размером буффера, по-дефолту когда он в двое превышает размер начально выделенный для хранения объектов - происходит sweep

#### Сборщик мусора запускается по трем причинам:

1. **Превышение динамического лимита памяти**, установленного переменной **`GOGC`** (по умолчанию 100).
2. **Прошло 2 минуты** без GC (за это отвечает фоновый тред **Sysmon**). Это можно отключить, установив `GOGC < 0`.
3. **Ручной запуск** с помощью `runtime.GC()`
#### цифры

сборщик мусора забирает около 20% времени работы больших приложений 90% из которых sweep стадия, ее улучшают с каждой новой итерацией сборщика с помощью крутых инструкций работы с памятью

#### Профилирование

это процесс измерения того, **как программа использует ресурсы**

1. gctrace - самый простой встроенный инструмент для профилирования
 ```bash
GODEBUG=gctrace=1 go run main.go
```
Чисто быстро посмотреть, как часто и как долго работает GC

2. pprof - **встроенный профилировщик Go**, который собирает и визуализирует статистику работы программы во время выполнения.
добавим в код
```go
import _ "net/http/pprof"
import "net/http"

func main() {
    go func() {
        http.ListenAndServe("localhost:6060", nil)
    }()
    runServer()
}
```
и уже чтобы посмотреть
```
http://localhost:6060/debug/pprof/
```

2.  есть другие, ну понятно дальше 

|Симптом|Что значит|Что делать|
|---|---|---|
|GC запускается слишком часто|Heap растёт быстро, GOGC слишком низкий|Повысить `GOGC`|
|Долгие паузы STW|слишком большой heap или активная аллокация|Оптимизировать память, уменьшить временные объекты|
|Высокий `GCCPUFraction` (>0.25)|GC ест много CPU|Реже запускать GC или переработать логику|
|Малый freed-memory gain|объекты удерживаются ссылками|Проверить утечки или глобальные кэши|
