
Go Heap Allocator основан на идеях **TCMalloc (Thread-Caching Allocator)**, главная из которых — **многоуровневое представление памяти** и использование **отдельных, неблокируемых (lock-free) блоков памяти для каждого потока (thread)**.

---

### Арены (Arenas) и Страницы (Pages)

Вместо того чтобы запрашивать у операционной системы (ОС) небольшие, частые фрагменты памяти, что может вызвать падение производительности, Go:

1. Запрашивает у ОС сразу **огромные непрерывные куски памяти**, называемые **аренами**.
2. Размер арены зависит от платформы. Например, на 64-битном Linux арена обычно имеет размер **64 MiB**.

|**Платформа**|**Размер арены**|
|---|---|
|linux/amd64|64 MiB|
|windows/amd64|4 MiB|
|32-bit|4 MiB|

Как только арена получена, она сразу же делится на более мелкие, фиксированные блоки — **страницы (pages)** размером **8 KiB**.

_Арены_ и _страницы_ — это просто **вшитые константы** и не несут никакой метаинформации. Управлением всей памятью (аренами и страницами) занимается основная структура аллокатора — **mheap**.

---

### Проблема Внешней Фрагментации

Фрагментация памяти возникает, когда свободное пространство становится раздробленным на множество мелких, несмежных блоков.

- **Внешняя Фрагментация:** Происходит, когда общее количество свободной памяти достаточно для выделения нужного объекта, но эта память **расположена несмежно** (не подряд). **Пример:** Вы освобождаете два блока по 2 байта. В сумме у вас есть 4 свободных байта. Но поскольку эти два блока разделены занятой памятью (они "в разных местах"), вы **не можете** выделить один объект размером 4 байта.  
    

---

### Решение Фрагментации: Пулы Фиксированных Размеров

Для борьбы с внешней фрагментацией разработчики Go ввели механизм, основанный на анализе **часто используемых размеров переменных**.

Go создает **пулы памяти фиксированных размеров**, предназначенные для наиболее частых размеров объектов, используемых программистами. Это гарантирует, что для большинства небольших объектов всегда будет готовый блок подходящего размера, что сводит к минимуме появление несмежных свободных участков.

---

## Пулы Объектов: `mspan` и `mcentral`

Go Heap Allocator борется с внешней фрагментацией, выделяя память из арен в виде **пулов фиксированных размеров** (`mspan`), которые затем управляются структурами `mcentral`.

### 1. `mspan` (Минимальный Юнит Памяти с Метаинформацией)

`mspan` — это минимальный блок памяти, выделенный из арены, который имеет **метаинформацию** о хранящихся в нем объектах. Он служит пулом для объектов одного фиксированного размера.

#### Классы Размеров (`spanclass`)

Каждый `mspan` имеет поле **`spanclass`**, которое определяет размер объектов, для которых он создан. Всего в Go **67 классов** размеров.

|**class**|**bytes/obj**|**bytes/span**|**objects**|**tail waste**|**max waste**|
|---|---|---|---|---|---|
|1|8|8192|1024|0|87.50%|
|2|16|8192|512|0|43.75%|
|3|24|8192|341|8|29.24%|
|4|32|8192|256|0|21.88%|
|...|...|...|...|...|...|
|67|32768|32768 (4 стр.)|1|0|12.50%|

#### Поля Структуры `mspan`

`mspan` — это двусвязный список, который содержит критическую метаинформацию:

```go
type mspan struct {
    next *mspan     // Следующий span в списке
    prev *mspan     // Предыдущий span в списке
    
    startAddr uintptr // Адрес начала пула на арене
    npages    uintptr // Количество страниц (по 8KiB) в пуле
    
    spanclass spanClass // Класс размера пула
    freeindex uint16  // Индекс, с которого начинать поиск свободного блока (для быстрого поиска)
    nelems uint16     // Общее количество объектов в пуле
    
    allocCount uint16 // Количество выделенных (занятых) объектов
    allocBits *gcBits // Битовая карта занятых/свободных ячеек (для аллокатора и GC)
    gcmarkBits *gcBits // Битовая карта для Garbage Collector

    // ... другие поля
}
```

- **`freeindex` и `allocBits`**: Работают вместе для **моментального поиска свободного блока** (механизм быстрого пути). `allocBits` — это битовая карта, где каждый бит показывает, занят объект или свободен.  
    

### 2. Внутренняя Фрагментация (`Waste`)

Поскольку `mspan` имеет фиксированный размер (кратный страницам 8 KiB), не все размеры объектов делят его без остатка. Это приводит к **внутренней фрагментации**:

- **Tail Waste (Остаток)**: Неиспользуемый остаток памяти в конце `mspan`, когда размер пула (`bytes/span`) не делится нацело на размер объекта (`bytes/obj`). _Например, для класса 3 (объект 24b) теряется 8 байт (`tail waste 8`)._  
    
- **Max Waste (Максимальные потери)**: Указывает, какой процент памяти в худшем случае может быть потерян. Это гарантирует, что для любого класса объектов **потери не превысят 50%**. _Например, для класса 5 (объект 48b) программисты гарантированно теряют 31% памяти в худшем случае, чтобы обеспечить быстрые аллокации._  
    

---

### 3. `mcentral` (Управление Пулами)

Структура `mcentral` отвечает за управление всеми `mspan` **одного и того же класса размеров**.

- **Количество:** В Go существует 67 классов размеров, следовательно, существует **67 структур `mcentral`**.  
    
- **Назначение:** `mcentral` действует как центральный диспетчер, снабжающий потоки (`mcache`) необходимыми `mspan`.  
    

#### Структура `mcentral` (Управление Жизненным Циклом)

```go
type mcentral struct {    
	spanclass spanClass    
	partial [2]spanSet // Список mspan, в которых есть свободные объекты    
	full    [2]spanSet // Список mspan, в которых нет свободных объектов}
```

- **`spanclass`**: Определяет, какому классу принадлежит данный `mcentral` и какие `mspan` он содержит.  
    
- **`partial` (Ранее `nonempty`)**: Содержит список `mspan`, в которых есть **хотя бы один свободный** блок. `mcentral` выдает `mspan` из этого списка.  
    
- **`full` (Ранее `empty`)**: Содержит список `mspan`, в которых **нет свободных** блоков (все заняты).  
    

В течение работы программы `mspan` перемещаются между списками `partial` и `full`: если в `mspan` освобождается объект, он может быть перемещен из `full` в `partial`; если все объекты в `mspan` выделены, он переходит из `partial` в `full`.

## Go Concurrency Model и Thread-Local Caching (mcache)

Для решения проблемы **Race Condition** (состояние гонки) при одновременном доступе нескольких потоков ОС (M) к общей куче (heap) и для избежания **падения производительности** из-за использования глобальных блокировок (`lock`), Go использует механизм **локального кэширования для каждого потока** (`mcache`).

---

### Структура `mcache`

**`mcache`** — это структура, которая предоставляет **отдельный блок памяти** для каждого **потока ОС (M)**. Поток работает со своим `mcache` **без необходимости введения блокировок**, что делает выделение небольших объектов очень быстрым (т.е., **lock-free**).

### Локальные `mspan`

`mcache` содержит массив **локальных `mspan`**, которые используются для выделения объектов.

Ключевой элемент `mcache` — массив **`alloc`**:

- `alloc` — это массив, где каждый элемент представляет собой двусвязный список `mspan`.  
    

#### Оптимизация для Сборщика Мусора (GC)

Массив `alloc` состоит всего из двух элементов, что является оптимизацией для ускорения работы GC:

1. **`alloc[0]` (NoScan Spans):** Содержит двусвязный список `mspan` для объектов, которые **не содержат указателей** (например, простой `int`, `float`, или срез `[]byte` без указателей внутри).
2. **`alloc[1]` (Scan Spans):** Содержит двусвязный список `mspan` для объектов, которые **содержат указатели** (например, `*int`, `struct` с полями-указателями).

**Цель:** Разделяя объекты на "сканируемые" (`scan`) и "несканируемые" (`noscan`), GC может полностью **пропустить сканирование** содержимого `mspan` в списке `alloc[0]`. Это значительно ускоряет сборку мусора, избегая лишнего обхода (`traverse`).

---

### Рабочий Поток

1. Когда горутина (G) запрашивает небольшой объект, поток (M), на котором она работает, сначала обращается к **своему `mcache`**.
2. `mcache` извлекает `mspan` соответствующего **класса размера** (например, для 16 байт) из своего массива `alloc`.
3. Выделение памяти происходит прямо из этого локального `mspan` **без блокировок**, что обеспечивает максимальную производительность.
4. Если локальный `mcache` пуст, он обращается к **`mcentral`** (который управляет пулами для данного класса), чтобы получить новый `mspan` (этот шаг уже требует синхронизации).
