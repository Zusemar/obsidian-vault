
#### **Горутина** — это 
**функция, выполняющаяся конкурентно** с другими горутинами в том же адресном пространстве.

Более детально её суть раскрывается через следующие характеристики:

- **Легковесность:** Горутины часто называют **«легковесными потоками»**, потому что их стоимость крайне мала — практически она сводится к выделению небольшого объема памяти под стек.

- **Динамические стеки:** В отличие от потоков ОС, стеки горутин начинаются с малого размера и **растут (или уменьшаются) в куче** по мере необходимости, что позволяет запускать миллионы горутин на ограниченных ресурсах.

- **Управление в пространстве пользователя:** Горутины управляются **средой выполнения (runtime) Go**, а не ядром операционной системы. Планировщик Go мультиплексирует (распределяет) множество горутин на меньшее количество системных потоков (модель M:N).

- **Простота жизненного цикла:** Новая горутина создается простым добавлением ключевого слова **go** перед вызовом функции или метода. Когда функция завершается, горутина **выходит автоматически и бесшумно**.

- **Изоляция и взаимодействие:** Горутины спроектированы так, чтобы не делить память напрямую через блокировки, а взаимодействовать через **каналы**, что предотвращает состояние гонки.

Если подытожить: это **единица конкурентности**, которая позволяет структурировать программу как набор независимо выполняющихся процессов, эффективно использующих аппаратные ресурсы

1. Параметры стека
В коде рантайма создание новой горутины инициируется функцией **runtime.newproc**.

Когда компилятор встречает в вашем коде ключевое слово `go` (например, `go func(...)`), он преобразует этот вызов в обращение к этой внутренней функции рантайма. Именно `runtime.newproc` отвечает за создание новой структуры горутины, выделение ей стека и помещение её в очередь на исполнение

Управление памятью стека является критически важным для легковесности горутин:

- **stack**: структура, описывающая фактическую память стека в диапазоне `[lo, hi)`.
- **stackguard0**: указатель стека, который проверяется в прологе функции для определения необходимости роста стека. Обычно равен `stack.lo + StackGuard`, но может принимать значение `StackPreempt` для инициирования вытеснения горутины.
- **stackguard1**: аналогичный указатель, используемый в прологе системного стека (`//go:systemstack`).

2. Состояние и контекст выполнения

Для переключения между задачами рантайм сохраняет состояние горутины в специальной вспомогательной структуре:

- **gobuf**: содержит сохраненные регистры, такие как указатель стека (`sp`), счетчик команд (`pc`), указатель на саму горутину (`g`), контекст (`ctxt`), а также регистры `lr` и `bp` для соответствующих архитектур.
- **guintptr**: специальный тип для хранения указателя на горутину в виде `uintptr`, что позволяет обходить барьеры записи (write barriers) при работе планировщика.

3. Обработка ошибок и отложенные вызовы

Горутина сама отслеживает свои цепочки паник и отложенных функций:

- **_defer**: указатель на список отложенных вызовов (`defer`). Эти структуры могут располагаться как на стеке, так и в куче.
- **_panic**: указатель на информацию об активной панике. Эти данные всегда живут только на стеке горутины.

4. Синхронизация и блокировки

Когда горутина не может продолжать выполнение, рантайм фиксирует причину и её место в очередях:

- **waitReason**: поле, объясняющее, почему горутина была остановлена (например, ожидание в канале — `waitReasonChanReceive`, блокировка мьютекса — `waitReasonSyncMutexLock` или ожидание ввода-вывода — `waitReasonIOWait`).
- **sudog**: вспомогательная структура (pseudo-g), которая представляет горутину в списках ожидания (например, при отправке или получении данных из канала). Это необходимо, так как отношение между горутинами и объектами синхронизации может быть «многие ко многим».

5. Дополнительная информация

- **ancestorInfo**: может содержать детали о том, где была запущена данная горутина (ID родительской горутины, PC инструкции `go`).
- **atomic.Uint64 goidgen**: используется в системе для генерации уникальных идентификаторов горутин (`goid`)
#### Cпособы синхронизации горутин через каналы

1. Синхронизация через небуферизированные каналы

Небуферизированные каналы объединяют передачу данных с гарантированной синхронизацией: они обеспечивают переход двух горутин в известное состояние в момент обмена.

- **Ожидание завершения:** Если горутина выполняет задачу в фоне, она может отправить сигнал в канал по окончании работы. Основная горутина, пытаясь прочитать из этого канала, будет заблокирована до тех пор, пока сигнал не поступит.
- **Блокировка:** В небуферизированном канале отправитель блокируется до тех пор, пока получатель не заберет значение. И наоборот, получатель всегда блокируется, пока данных в канале нет.
- **Гарантии памяти:** Согласно модели памяти Go, операция отправки в канал всегда происходит **до** завершения соответствующей операции приема в другой горутине.

2. Использование буферизированных каналов как семафоров

Каналы с буфером позволяют ограничить количество одновременно выполняемых задач (пропускную способность).

- **Модель семафора:** Емкость буфера канала соответствует максимальному количеству одновременных процессов. Отправка значения в канал (заполнение буфера) трактуется как «захват» семафора, а прием (освобождение места в буфере) — как его «отпуск».
- **Ограничение ресурсов:** Если буфер заполнен, любая новая горутина, пытающаяся отправить данные в канал, будет заблокирована, пока один из текущих обработчиков не завершит работу и не освободит место.

3. Сигнализация через закрытие канала

Закрытие канала (`close`) является мощным инструментом широковещательной синхронизации.

- **Уведомление о завершении:** Операция закрытия канала синхронизирована с приемом, который возвращает нулевое значение типа канала. Это часто используется для того, чтобы сообщить рабочим горутинам о необходимости прекратить выполнение.
- **Context:** Пакет `context` активно использует этот механизм — метод `ctx.Done()` возвращает канал, который закрывается при отмене контекста или истечении таймаута, сигнализируя горутине о необходимости очистки ресурсов и выхода.

4. Динамическая координация (каналы каналов)

Каналы в Go являются значениями первого класса, их можно передавать через другие каналы. Это позволяет создавать сложные структуры, такие как **безопасное параллельное демультиплексирование**.

- Например, в системе RPC клиент может передать в объекте запроса канал для ответа. Сервер обработает запрос и отправит результат именно в тот канал, который предоставил клиент, обеспечивая изолированную и безопасную передачу данных без использования мьютексов.



#### Рост стека горутины

Максимальный стэк горутины начинается с **1 мб**:

var maxstacksize uintptr = 1 << 20 // enough until runtime.main sets it for real

Затем, после запуска main горутина выставляет его в **1 гб** на 64-битных системах или **250 мб** на 32-битных:
```go 
// Max stack size is 1 GB on 64-bit, 250 MB on 32-bit.  
// Using decimal instead of binary GB and MB because  
// they look nicer in the stack overflow failure message.  
if goarch.PtrSize == 8 {  
   maxstacksize = 1000000000  
} else {  
   maxstacksize = 250000000  
}
```

Рост стека

Стек горутины увеличивается в 2 раза до порога максимального размера стека:
```go
// Allocate a bigger segment and move the stack.
oldsize := gp.stack.hi - gp.stack.lo
newsize := oldsize * 2

```

Переполнение стек

Если стек переполняется, то мы получаем ошибку рантайма stack overflow:
```go
if newsize > maxstacksize || newsize > maxstackceiling {
    if maxstacksize < maxstackceiling {
      print("runtime: goroutine stack exceeds ", maxstacksize, "-byte limit\n")
    } else {
      print("runtime: goroutine stack exceeds ", maxstackceiling, "-byte limit\n")
    }
    print("runtime: sp=", hex(sp), " stack=[", hex(gp.stack.lo), ", ", hex(gp.stack.hi), "]\n")
    throw("stack overflow")
  }
```


#### **Утечка горутин** (goroutine leak)

— это состояние, при котором горутина продолжает выполняться или остается заблокированной в памяти даже после того, как результат её работы перестал быть нужен основной программе. Поскольку горутины управляются средой выполнения Go, а не операционной системой, сборщик мусора не может принудительно остановить заблокированную горутину, что приводит к постепенному и неограниченному росту потребления памяти.

Последствия утечек горутин

Игнорирование сигналов отмены может привести к серьезным проблемам:

- **Рост потребления памяти:** Каждая горутина занимает ресурсы стека, и при накоплении тысяч «зависших» горутин память может закончиться.
- **Утечка соединений:** Горутины могут продолжать удерживать открытые соединения с базами данных или сетевые сокеты, исчерпывая лимиты ресурсов.
- **Бесполезные вычисления:** Продолжение обработки запросов, которые уже были отменены пользователем, тратит циклы CPU.
- **Замедление завершения работы:** Программа может долго закрываться, ожидая завершения всех процессов

## Состояния горутины

| Состояние | Константа | Описание |
|---|---|---|
| Runnable | `_Grunnable` | В очереди, ждёт процессора |
| Running | `_Grunning` | Выполняется на M |
| Waiting | `_Gwaiting` | Заблокирована (канал, mutex, sleep) |
| Dead | `_Gdead` | Завершена, ожидает переиспользования |
| Syscall | `_Gsyscall` | Выполняет системный вызов |

## Мониторинг горутин

```go
// Количество живых горутин
n := runtime.NumGoroutine()
fmt.Printf("goroutines: %d\n", n)
// Увеличение без остановки → утечка!

// Stack trace всех горутин (для отладки)
buf := make([]byte, 1<<20)
n = runtime.Stack(buf, true)  // true = все горутины
fmt.Printf("%s", buf[:n])
```


## sync.WaitGroup — ожидание группы горутин

```go
var wg sync.WaitGroup

for _, url := range urls {
    wg.Add(1)
    go func(u string) {
        defer wg.Done()
        fetch(u)
    }(url)
}
wg.Wait()  // блокируется пока все Done не вызваны
```
